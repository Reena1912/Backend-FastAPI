# Concept: Cryptographic Hashing & Stateless Auth (JWT)

Security is one of the most critical responsibilities of a backend developer. In this module, we examine how to secure passwords and protect API resources using **Stateless JSON Web Tokens (JWT)**.

---

## 1. Password Hashing (Never store plaintext!)

If a database is compromised, storing plaintext passwords is catastrophic.
*   **Plaintext:** `"my_password123"`
*   **MD5/SHA256 (Incorrect):** Simply running a fast hash function makes the app vulnerable to **Rainbow Table** lookups (pre-computed lists of hashes).
*   **Key Derivation Functions (Correct):** Algorithms like Bcrypt, Argon2, or PBKDF2. They use a **salt** (random noise added to each password) and are *intentionally slow* to prevent brute-force attacks.

In our code, we use Python's built-in `hashlib.pbkdf2_hmac` to securely hash and verify passwords.

---

## 2. Stateful vs. Stateless Sessions

When a user logs in, how does the server remember them?

| Feature | Stateful (Sessions) | Stateless (JWT) |
| :--- | :--- | :--- |
| **Where is state stored?** | In the server's memory or database. | In the client's browser/device (inside the token). |
| **How does it work?** | Server gives client a `session_id`. Client sends it with every request. Server looks it up in DB. | Server signs a JSON payload containing user data and returns it. Client sends token. Server decodes and verifies the signature. |
| **Scalability** | Harder: Multiple servers need a shared session store (like Redis). | Easier: Any server can verify the token signature without hitting a database. |

---

## 3. How JSON Web Tokens (JWT) Work

A JWT is a string divided into three parts separated by dots (`.`):

```
[ Header ] . [ Payload ] . [ Signature ]
```

1.  **Header (Base64 Encoded):** Metadata about the token (e.g., hash algorithm used, token type).
2.  **Payload (Base64 Encoded):** The data (claims) you want to store. e.g., `{"user_id": 42, "role": "admin", "exp": 1718468000}`.
3.  **Signature:** A cryptographic seal. It is generated by taking the encoded Header and Payload, combining them with a secret key known *only* to the server, and running them through a hashing algorithm (like HMAC-SHA256).

```
signature = HMAC_SHA256(header + "." + payload, SECRET_KEY)
```

### The Security Rule:
Anyone can read the payload of a JWT (it is just Base64 encoded). However, **no one can modify the payload** without breaking the signature, because they don't have your `SECRET_KEY`. If a hacker edits `role: user` to `role: admin`, the signature verification will fail, and the server rejects the request.
